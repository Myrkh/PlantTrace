Démarrage en 7 minutes

Le chemin normal pour un nouveau projet est volontairement court. Préparer le corpus, vérifier la couverture, chercher ou extraire.

Premier lancement

  1. Ouvrir PlantTrace et choisir le dossier d'index local.
  2. Choisir le dossier source contenant les PDF.
  3. Cliquer Indexer et attendre le rapport final.
  4. Ouvrir Couverture pour contrôler pages texte, OCR requis et échecs.
  5. Aller dans Recherche et tester une référence connue : FV1100, JDY, 10-P-12345.

Règle de lecture

  1. Un résultat positif doit être lu avec son PDF, sa page et son extrait.
  2. Un score flou aide à prioriser, il ne prouve pas seul.
  3. Une extraction est pilotée par les règles projet actives.
  4. Un export est un livrable de travail, pas une vérité documentaire absolue.
  5. Si des pages ne sont pas OCRisées, qualifier explicitement l'absence.

Workflow recommandé

Ce workflow évite la confusion entre recherche, extraction et livrable. La preuve reste au centre à chaque étape.

La stoplist est une liste de valeurs à ignorer globalement (une par ligne). Fiche de référence

La fiche consolide en un seul écran tout ce que PlantTrace sait sur une référence précise : où elle apparaît, avec quoi elle est associée, et si elle est en conflit ou a changé entre révisions.

Ce que la fiche construit

  1. Occurrences — toutes les pages où la référence est trouvée, classées par famille documentaire, avec chemin, page, type de match, trouvé comme, et extrait.
  2. Associations — les LINE, DOC et autres TAG présents sur les mêmes pages, représentant des relations probables. Triées par type (LINE en premier).
  3. Alertes — conflits actifs (valeurs contradictoires) et changements de révision (ajout/retrait/modification) affectant cette référence.

Comment l'utiliser

  1. Saisir la référence dans le champ de la fiche (onglet Fiche de l'activité Recherche).
  2. La fiche cherche en mode hybrid avec une limite de 500 résultats pour être exhaustive.
  3. Lire les associations : elles révèlent quels documents et lignes sont probablement liés à ce TAG.
  4. Vérifier les alertes : une alerte conflict indique des valeurs contradictoires à arbitrer.
  5. Exporter la fiche en XLSX pour la revue projet.

Familles documentaires

PlantTrace classe automatiquement chaque PDF indexé dans une famille projet en analysant son nom de fichier et ses trois premières pages. La classification est reproductible et explicite : chaque décision est accompagnée d'une preuve textuelle.

Algorithme

Pour chaque PDF : score += 4 si un terme est trouvé dans le nom de fichier, score += 3 si un terme est trouvé dans les trois premières pages (normalisés en majuscules, tirets et underscores remplacés par des espaces). Le meilleur score détermine la famille. Score < 3 → classé A classer.

Conflits et Révisions

Deux outils de contrôle documentaire : Conflits détecte les incohérences dans un corpus existant ; Révisions compare deux lots de PDF pour mesurer ce qui a changé.

Conflits — Détection d'incohérences

Pour chaque TAG extrait, PlantTrace collecte les LINE, DOC et DESCRIPTION associés (co-présents sur la même page ou dans le même extrait). Si un même TAG apparaît avec des valeurs différentes pour un même champ, c'est un conflit.

High
Conflit LINE

Un TAG est associé à deux lignes de tuyauterie différentes dans des documents distincts. Exemple : FV1100 associé à 6-HC-12345 dans un PID et à 8-HC-12345 dans un loop diagram. À arbitrer impérativement.

Medium
Conflit DOC

Un TAG est référencé avec des numéros de document contradictoires. Peut indiquer un document renommé, une erreur de transcription, ou un TAG dupliqué sur deux isométriques différentes.

Low
Conflit DESCRIPTION

Libellé ou désignation du TAG différent entre documents. Souvent dû à des abréviations variables (Control valve vs Vanne de contrôle). À vérifier mais rarement bloquant.

Révisions — Comparaison old/new

Comparaison de deux dossiers PDF représentant l'ancienne et la nouvelle révision du corpus. PlantTrace ré-indexe les deux dossiers dans des sous-projets isolés, applique les règles actives et compare les inventaires.

Workflow révision

  1. Pointer Ancien dossier PDF vers la révision précédente.
  2. Pointer Nouveau dossier PDF vers la révision courante.
  3. Lancer la comparaison. PlantTrace indexe les deux corpus dans .planttrace/revisions/old et new.
  4. Lire les changements : added, removed, modified.
  5. Exporter pour la revue projet.
Statut added

Référence présente en nouvelle révision uniquement. Nouvel équipement ou nouvelle ligne.

Statut removed

Référence présente en ancienne révision uniquement. Équipement supprimé ou numéro modifié.

Statut modified

Référence conservée mais preuve, page ou extrait différent. La localisation documentaire a changé.

Signature de changement

Deux occurrences sont considérées identiques si elles partagent le même chemin relatif (normalisé), la même page et le même extrait compact (après suppression des espaces multiples). Une différence sur l'un de ces trois critères produit un statut modified.

Confiance et preuve

PlantTrace est volontairement strict : il indique ce qu'il a vu, où il l'a vu, et dans quelles conditions de couverture.

OCR et absence

Si une page est scannée sans texte et que l'OCR n'a pas tourné ou a échoué, PlantTrace ne peut pas prouver qu'une référence est absente de cette page. Le statut ocr_required ou ocr_failed est reporté dans chaque résultat.

Avant de conclure "non trouvé", vérifier Couverture et exporter le triage OCR si nécessaire.

Vectoriel / sémantique

La recherche vectorielle peut aider sur des idées floues comme "fournisseur café". Pour les TAG, lignes et numéros documentaires, la preuve doit rester déterministe : normalisation, exact match, règles et extrait. Le mode semantic nécessite un modèle SentenceTransformers placé localement dans .planttrace/models/embedding-model.

Statut pageSignificationPreuve valide ?
okPage avec texte natif extrait normalement.Oui
ocr_okPage scannée, OCR exécuté avec succès.Oui (qualité OCR dépendante)
ocr_requiredPage scannée, aucun texte extrait, OCR non lancé.Non — absence non qualifiable
ocr_failedOCR lancé mais a échoué sur cette page.Non — absence non qualifiable

Mode opératoire détaillé

Notes courtes pour rappel pendant l'utilisation.

Recherche ponctuelle

Taper la référence, choisir le mode, cliquer Chercher. Les résultats sont triés par priorité : exact normalisé en premier, puis confusion OCR normalisée, puis BM25, puis sémantique, puis flou. Double-cliquer ouvre le PDF source à la bonne page si le visualiseur est disponible.

Pour une référence industrielle (FV1100, 10-P-12345) : privilégier le mode exact ou hybrid. Pour un nom propre ou une phrase : privilégier text ou semantic.

Extraction projet

S'assurer que les règles sont actives (colonne "Active" à "oui") et sauvegardées. Cliquer Extraire. Les colonnes importantes sont Type, Valeur, Règle, Fichier, Page et Confiance.

Les valeurs présentes dans la stoplist sont silencieusement exclues. Si un candidat inattendu ressort trop souvent, l'ajouter à la stoplist et relancer.

Batch depuis un fichier

Le batch accepte une saisie directe (une référence par ligne, ou séparées par virgules/points-virgules) ou l'import d'un fichier XLSX, CSV ou TXT via le bouton Charger fichier.

Pour un XLSX, PlantTrace cherche automatiquement une colonne dont le header est tag, reference, ref, query, value ou valeur. Si aucun header reconnu n'est trouvé, il prend la première colonne non vide. Les doublons sont dédupliqués (insensible à la casse).

Conflits et révisions

Conflits opère sur le corpus actuel indexé. Il extrait toutes les références, collecte les preuves d'association (LINE, DOC, DESCRIPTION co-présents avec chaque TAG), et signale les cas où plusieurs valeurs différentes sont associées au même champ pour un même TAG.

Révisions ré-indexe entièrement deux dossiers PDF dans des projets temporaires isolés (.planttrace/revisions/old et new). Les dossiers précédents sont purgés avant chaque lancement. Les règles actives du projet principal sont appliquées aux deux corpus.

Templates de tableau

Le Registre TAG produit une ligne par TAG unique trouvé dans l'extraction. Chaque ligne consolide : description probable (issue des proximitès textuelles), lignes associées, numéros de documents associés, familles documentaires sources, pages sources, nombre de preuves, résumé des conflits actifs et des changements de révision.

Limite par champ : 6 valeurs maximum par colonne (suffixe … si tronqué). Source_pages est limitée à 8 pour la lisibilité.

Normalisation et gestion OCR

PlantTrace ne cherche jamais exactement la chaîne saisie. Il produit plusieurs variantes normalisées pour absorber les conventions typographiques et les confusions OCR courantes.

Variantes exactes générées

Pour la requête FV1100, PlantTrace teste :

  • FV1100 — forme brute.
  • FV1100 compact (diacritiques retirés, majuscules).
  • FV-1100 — avec tiret.
  • FV 1100 — avec espace.
  • FV_1100 — avec underscore.

Confusions OCR absorbées

Les OCR confondent fréquemment :

  • O ↔ 0 (lettre O / zéro)
  • I ↔ 1 (lettre I / un)
  • l ↔ 1 (L minuscule / un)

Le match_type ocr_confusion_normalized indique que la référence a été trouvée après application de ces substitutions.

Détection automatique d'identifiant

Le mode auto appelle is_probable_identifier() : si la requête compacte (sans ponctuation) contient à la fois des lettres et des chiffres et fait au moins 3 caractères, elle est traitée comme un identifiant industriel (exact en premier). Sinon, elle part en recherche texte puis floue.

Modes de recherche en détail

ModeMécanismeCas d'usagePriorité résultat
exact Cherche toutes les variantes normalisées (tiret, espace, underscore, sans séparateur) et les variantes avec confusion OCR dans la base SQLite. TAG, numéro de ligne, numéro documentaire, initiales. Meilleur choix pour les identifiants industriels. 1 (exact_normalized), 2 (ocr_confusion_normalized)
text Recherche plein texte BM25 via SQLite FTS5. Tokenisation sur les caractères alphanumériques. Mots, phrases, noms propres, tout ce qui n'est pas un code structuré. 3 (text_bm25)
fuzzy Tolérance aux fautes de frappe (distance d'édition). Plus lent, résultats moins précis. Référence dont l'orthographe exacte est incertaine. 5
semantic Embeddings vectoriels locaux (SentenceTransformers). Requiert un modèle dans .planttrace/models/embedding-model et une vectorisation préalable. Recherche conceptuelle ("fournisseur de café", "vanne de régulation pression"). 4 (semantic)
hybrid Lance exact + text + semantic en parallèle. Si aucun résultat, déclenche fuzzy. Choix par défaut. Couvre tous les cas sans configuration. Selon le type de chaque hit
auto Si la requête ressemble à un identifiant (lettres + chiffres) : exact d'abord, puis text, puis fuzzy. Sinon : text puis fuzzy. Ligne de commande CLI sans choix de mode explicite. Selon le type de chaque hit

Interface en ligne de commande

PlantTrace expose une CLI complète pour intégration dans des scripts, pipelines CI ou automatisations PowerShell. Toutes les sorties JSON ou XLSX sont identiques à celles de l'interface graphique.

planttrace index — Indexer un dossier PDF 
# Indexation standard
planttrace index --project . --pdf-root "D:\Projet\PDF"

# Forcer la réindexation complète
planttrace index --project . --pdf-root "D:\Projet\PDF" --force

# Avec OCR Tesseract pour les pages scannées
planttrace index --project . --pdf-root "D:\Projet\PDF" --ocr --ocr-lang fra

Flags : --project (dossier racine projet), --pdf-root (dossier PDF source), --force (réindexe même les fichiers déjà vus), --ocr (active Tesseract), --ocr-lang (langue Tesseract, défaut : eng).

Sortie JSON : {"indexed": N, "skipped": N, "failed": N}. Code retour 0 si zéro échec, 1 sinon.

planttrace search — Recherche une référence 
# Recherche hybride, sortie texte
planttrace search --project . --query FV1100

# Mode exact, export XLSX
planttrace search --project . --query FV1100 --mode exact --output results.xlsx

# Sortie JSON pour traitement script
planttrace search --project . --query FV1100 --mode hybrid --json --limit 100

Flags : --mode (auto, exact, text, fuzzy, semantic, hybrid), --limit (défaut : 50), --output (chemin XLSX), --json (sortie JSON stdout).

Code retour 0 si au moins un résultat non "not_found", 1 sinon.

planttrace extract — Extraction par règles 
# Extraction avec sortie texte
planttrace extract --project .

# Export XLSX avec limite augmentée
planttrace extract --project . --output extraction.xlsx --limit 50000

# JSON pour pipeline
planttrace extract --project . --json --limit 10000

Utilise les règles actives dans .planttrace/rules.json. Flags : --limit (défaut : 10 000), --output, --json.

planttrace coverage — Rapport de couverture 
planttrace coverage --project .

Sortie JSON : nombre total de documents, pages, pages texte, pages OCR requises, pages OCR échouées. Utile pour qualifier une absence dans un rapport.

planttrace embed / semantic-status — Vectorisation 
# Vectoriser le corpus (modèle local requis)
planttrace embed --project .

# Vérifier l'état de la vectorisation
planttrace semantic-status --project .

Le modèle SentenceTransformers doit être placé dans .planttrace/models/embedding-model avant de vectoriser. semantic-status retourne {"available": true/false, "chunk_count": N, ...}. Code retour 0 si disponible.

Structure du dossier projet

Tout ce que PlantTrace écrit est contenu dans un sous-dossier .planttrace/ à la racine du projet choisi. Le dossier PDF source n'est jamais modifié.

.planttrace/ ├── planttrace.sqlite — Base principale (documents, pages, textes, index FTS5) ├── embeddings.npy — Vecteurs sémantiques (numpy, généré par `embed`) ├── rules.json — Règles projet actives (format pack v1) ├── stoplist.txt — Valeurs à ignorer globalement └── revisions/ ├── old/ │ └── planttrace.sqlite — Sous-projet ancienne révision (temporaire) └── new/ └── planttrace.sqlite — Sous-projet nouvelle révision (temporaire)
Portabilité

Le dossier .planttrace/ peut être copié avec le projet. Le chemin absolu des PDF est stocké dans la base ; si le corpus est déplacé, relancer planttrace index --force pour reconstruire les chemins.

Distribution en entreprise

L'objectif est un lancement double-clic sans demander aux collègues de manipuler Python ou un terminal.

Objectif packaging

Un dossier ou installeur Windows contenant l'application, les dépendances Python gelées (via PyInstaller ou cx_Freeze), les assets, la documentation HTML locale et les commandes nécessaires. Le runtime d'indexation et de recherche reste entièrement local. L'interface graphique (PySide6) est incluse dans le bundle.

OCR et Tesseract

Tesseract n'est pas inclus dans le bundle Python. Pour les sites qui en ont besoin, distribuer l'installeur Tesseract Windows séparément et documenter le chemin attendu. PlantTrace détecte Tesseract automatiquement dans le PATH système.
